# Doripe Notify Character Algorithm Design

## 목적

`/notify`의 사진 선택 테스트를 공유/바이럴용 장소 취향 캐릭터 결과로 만든다.

궁합 점수는 다음 단계로 두고, 이번 범위는 10번의 사진 선택을 통해 캐릭터 결과까지 안정적으로 산출하는 것이다.

## 현재 상태

- 현재 `computeCharacter`는 B 선택이 5개 이상이면 `route_planner`, 아니면 `quiet_collector`를 반환한다.
- 현재 사진 선택 화면은 같은 2개 이미지를 반복 사용한다.
- 현재 캐릭터는 2개뿐이다.

## 결정된 방향

- 목적: 공유/바이럴용
- 결과 톤: 장소 분위기 기반, 말투는 공유하고 싶은 문장
- 문항: 2장 중 1장 선택을 10번
- 이미지 수: 20개
- 캐릭터 수: 6개
- 알고리즘: 캐릭터 점수 + 취향축 타이브레이크 혼합 방식

## 캐릭터 6개

### 햇살 아래 오래 앉아있는 사람

- 분위기: 조용한 창가, 식물, 낮, 오래 머무는 공간
- 태그: 창가, 조용함, 낮
- 점수 키: `sunny_window`

### 골목만 보면 일단 꺾는 사람

- 분위기: 로컬 골목, 작은 가게, 산책, 우연한 발견
- 태그: 골목, 산책, 로컬
- 점수 키: `alley_wanderer`

### 간판 예쁘면 이미 반쯤 넘어간 사람

- 분위기: 외관, 편집샵, 트렌디한 거리, 도시적인 장면
- 태그: 외관, 트렌디, 도시
- 점수 키: `city_aesthetic`

### 맛없는 감성은 용서 못 하는 사람

- 분위기: 음식, 브런치, 파스타, 실패 없는 메뉴
- 태그: 맛집, 메뉴, 실패없음
- 점수 키: `food_first`

### 조명 어두워지면 취향 살아나는 사람

- 분위기: 와인바, 저녁, 낮은 조명, 대화, 데이트
- 태그: 밤, 데이트, 무드
- 점수 키: `night_mood`

### 하루를 한 장면으로 엮는 사람

- 분위기: 카페-산책-식사, 자연스러운 동선, 하루 루트
- 태그: 루트, 동선, 하루
- 점수 키: `route_scene`

## 10개 사진 비교축

각 비교는 장소 무드와 사용 상황을 섞는다.

1. 조용한 창가 카페 vs 활기 있는 핫플
2. 한 장소 깊게 머물기 vs 여러 곳 이어 걷기
3. 감성 소품샵 vs 맛 중심 식당
4. 낮 브런치 vs 저녁 와인바
5. 익숙한 동네감 vs 새로운 발견감
6. 혼자 가기 좋은 곳 vs 친구와 가기 좋은 곳
7. 자연/초록 vs 도시/건축
8. 사진 예쁜 공간 vs 실패 없는 공간
9. 느린 감상 vs 가벼운 액티비티
10. 계획된 하루 vs 즉흥 발견

## 점수 방식

각 사진 선택지는 다음 값을 가진다.

- `characters`: 1~2개 캐릭터 점수
- `axes`: 1~3개 취향축 점수

예시:

```json
{
  "round": 1,
  "option": "A",
  "label": "조용한 창가 카페",
  "characters": {
    "sunny_window": 2,
    "alley_wanderer": 1
  },
  "axes": {
    "quiet": 2,
    "day": 1,
    "stay": 1
  }
}
```

최종 캐릭터는 캐릭터 점수 합산 1위로 결정한다.

동점이면 아래 순서로 결정한다.

1. 해당 캐릭터의 대표 축 점수
2. 마지막 3문항에서 더 많이 선택된 캐릭터
3. 기본 우선순위

## 기본 우선순위

동점이 완전히 풀리지 않을 때만 사용한다.

1. 햇살 아래 오래 앉아있는 사람
2. 골목만 보면 일단 꺾는 사람
3. 간판 예쁘면 이미 반쯤 넘어간 사람
4. 맛없는 감성은 용서 못 하는 사람
5. 조명 어두워지면 취향 살아나는 사람
6. 하루를 한 장면으로 엮는 사람

## 이미지 제작 기준

- 모든 이미지는 AI 이미지로 제작한다.
- 20개 이미지는 서로 다른 장소처럼 보여야 한다.
- 한 문항 안의 두 이미지는 시각적으로 명확히 달라야 한다.
- 이미지 안에 글자, 로고, UI 요소는 넣지 않는다.
- 장소 분위기만 보여준다.
- 너무 추상적인 무드컷보다 실제 방문하고 싶은 장소 사진처럼 만든다.

## 이미지 제작 시 주의점

질문 의도가 유저에게 바로 들키면 안 된다.

예를 들어 “혼자 가기 좋은 곳 vs 친구와 가기 좋은 곳”을 만들 때, 한쪽은 1인 테이블이고 다른 한쪽은 4인 테이블에 사람들이 앉아있는 식으로 만들면 안 된다. 유저가 “아, 지금 동행 인원 물어보는구나”라고 느끼면 테스트가 억지스러워진다.

이미지는 아래 방식으로 은은하게 차이를 만든다.

- 사람 수, 테이블 수, 지도, 화살표, 노골적인 간판, 데이트 소품 같은 직접 단서를 피한다.
- 두 선택지 모두 실제로 가고 싶어 보여야 한다.
- 한쪽이 명백히 더 예쁘거나 더 고급져 보이면 안 된다.
- 차이는 조명, 거리감, 시선 흐름, 공간 밀도, 소품 배치, 음식 비중, 창밖 풍경처럼 간접적으로 만든다.
- “루트/계획”은 지도나 동선을 보여주지 말고, 다음 장소로 이어질 것 같은 거리감이나 열린 문, 골목 깊이로 표현한다.
- “혼자/함께”는 인원수로 표현하지 말고, 좌석 간격, 테이블 위 여백, 공유하기 좋은 메뉴 구성처럼 간접적으로 표현한다.
- “맛/분위기”는 음식 유무로 나누지 말고, 음식이 화면의 주인공인지 공간의 무드가 주인공인지 정도만 다르게 한다.
- “핫플/로컬”은 사람 줄이나 SNS 촬영 장면으로 표현하지 말고, 외관의 선명함, 소재, 거리 분위기 차이로 표현한다.

좋은 기준은 “유저는 그냥 더 끌리는 사진을 골랐다고 느끼지만, 선택 데이터는 뒤에서 취향축으로 읽히는 것”이다.

## 구현 범위

이번 구현 범위:

- 20개 선택 이미지 등록
- 6개 캐릭터 데이터 등록
- 10라운드 선택 데이터 구조화
- 캐릭터 산출 알고리즘 구현
- 결과 화면에 6개 캐릭터 반영

이번 범위 제외:

- 친구와의 궁합 점수 고도화
- 실제 Doripe 장소 추천 알고리즘 연결
- 이미지 생성 자동화 파이프라인
